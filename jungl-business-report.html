Issue #1: Butterfly GIF Animations (1.6 MB)

Left_butterfly.gif: 985 KB · Right_butterfly.gif: 635 KB CRITICAL

The problem: Two animated butterfly GIFs are loaded on every page visit. GIF is the worst format for animations — it's from 1987. These two files alone are larger than many entire websites.

The fix (keeps your animations!):

  • Convert both GIFs to animated WebP — same animation, ~80% smaller file size. A 985KB GIF typically compresses to ~150-200KB as WebP.
  • Even better: convert to short looping MP4 videos with <video autoplay muted loop playsinline> — this can get them under 100KB each while looking smoother.
  • Add loading="lazy" so they don't load until the user scrolls to them.
  • Expected savings: 1,400 KB (cuts page weight by 35%)

How to do it: Use ezgif.com/gif-to-webp — upload each GIF, download the WebP version, upload to Shopify Files, update your theme code.

Issue #2: Render-Blocking CSS (459 KB)

bundle.css (384KB) + Judge.me CSS (75KB) block page rendering CRITICAL

The problem: The browser can't show ANYTHING to the user until it downloads and processes these two files. On a 3G mobile connection, that's 2-3 extra seconds of blank screen.

The fix:

  • Critical CSS inline: Extract the CSS needed for the first screen (above-the-fold) and put it directly in the HTML. Load the rest after the page renders. This alone could cut your First Contentful Paint by 1.5-2 seconds.
  • Judge.me CSS: Load it with media="print" onload="this.media='all'" — this makes it non-blocking. Reviews aren't visible on the first screen anyway.
  • Purge unused CSS: bundle.css is 384KB but your homepage probably uses only 30-40% of those styles. Use a tool like PurgeCSS to strip the unused rules.
  • Expected savings: 1.5-2 seconds faster First Contentful Paint

Issue #3: JavaScript Bloat (686 KB + 51 Inline Scripts)

bundle.js (387KB) + Lottie (299KB) + 51 inline scripts HIGH

The problem: Your page has 51 inline scripts plus two large JS files. The bundle.js is preloaded (good) but still blocks interactivity. The Lottie library is massive for a single eye animation.

The fix:

  • Replace lottie.min.js (299KB) with lottie-light (150KB) — the light version handles simple animations perfectly. That's a free 150KB saved.
  • Better yet: lazy-load Lottie — only load it when the eye animation scrolls into view. Use IntersectionObserver to trigger the download.
  • Defer bundle.js: Add defer attribute so it doesn't block parsing. Change from preload to normal loading with defer.
  • Audit inline scripts: 51 inline scripts suggests multiple apps injecting code. Each one of these blocks rendering. Identify which are from apps you actually use.
  • Expected savings: 200-300KB and 1-1.5 seconds faster Time to Interactive

Issue #4: Too Many Fonts (9 fonts, 262KB)

Gotham (3 weights) + Uniform (4 variants) + TT Travels + UniformProXcon HIGH

The problem: You're preloading 9 font files, all of which must download before text renders properly. One of them (TT Travels) is a TTF file (40KB) instead of the more compressed WOFF2 format — that's 2x larger than it needs to be.

The fix:

  • Reduce to 4-5 fonts max: Do you really need 4 Uniform variants? Pick 2 (Regular + Bold). Same for Gotham — Book + Bold is enough for most designs. This cuts 3-4 font downloads.
  • Convert TT-Travels-Next-Trial-Black.ttf to WOFF2: This single change saves ~20KB and loads faster. Use cloudconvert.com.
  • Use font-display: swap in your @font-face rules — shows text immediately in a fallback font while custom fonts load.
  • Expected savings: ~100KB and prevents invisible text during load

Issue #5: Third-Party App Bloat (6+ Apps Loading on Every Page)

Judge.me · Loloyal · SavedBy · Klaviyo · Clarity · CustomerFields HIGH

The problem: Each of these apps injects its own CSS and JavaScript into every single page load. Some are doing this even when they aren't visible or needed on that page. I found these loading on the homepage:

  • Judge.me (reviews): CSS + JS on every page, even where no reviews show
  • Loloyal (loyalty program): 39 references in your HTML — launcher widget loads everywhere
  • SavedBy (wishlist): 81 references — extremely heavy for a feature most visitors won't use
  • Klaviyo (email): Popup/tracking script on every page
  • Microsoft Clarity (analytics): Session recording — adds weight
  • CustomerFields (custom fields): Loading even for non-logged-in visitors

The fix:

  • SavedBy (81 references!): This is your biggest app offender. If wishlists aren't driving measurable revenue, remove it or replace with a lighter alternative.
  • Loloyal (39 references): Defer the loyalty launcher to load after page is fully visible. No one clicks a loyalty widget in the first 5 seconds.
  • Judge.me: Only load on product pages and collection pages — not homepage.
  • Clarity: Consider if you're actually using the session recordings. If not, remove it.
  • Expected savings: 1-2 seconds load time, 200-400KB

Issue #1: Subscribe-by-Default Checkout

⚠️ The Double-Edged Sword HIGH

The situation: When someone adds a product to cart, the default option is "Subscribe & Save (20% off)" rather than one-time purchase. You told me this generates your most support tickets because customers don't notice they're subscribing.

Why this is complicated:

  • Today, 100% of sales came through Recharge Subscriptions. Subscriptions ARE your revenue engine.
  • Your cohort data shows strong month-2 retention (10-29%), suggesting subscribers stick around.
  • BUT: customers who feel tricked don't leave good reviews, don't refer friends, and create support overhead.

My recommendation (nuanced):

  • Don't remove the subscribe option. It's working for retention and LTV.
  • DO make it more visible. Add a clear visual distinction — make "Subscribe & Save" a different colored card/box with explicit text: "Delivers every 30 days. Cancel anytime."
  • Make one-time purchase the default but show the subscription savings prominently: "Save 20% — subscribe for $X/month". Let people OPT IN to subscriptions rather than accidentally opting in.
  • Add a tooltip or info icon next to the subscribe option explaining exactly what it means.
  • Expected impact: You may lose 5-10% of subscription sign-ups but gain 15-25% more one-time purchases from people who were abandoning in confusion. Net positive.

Why Scaling Spend Hurts Conversions

Here's exactly what's happening when you increase spend and conversions get worse:

  1. You have 5 campaigns with 8-10+ ad sets total
  2. When you increase budget, Meta spreads the money across all these ad sets
  3. No individual ad set gets enough conversions to exit learning (needs 50/week)

The solution is counterintuitive: Spend MORE money on FEWER campaigns. Consolidate everything into 2 campaigns, each with 1-2 ad sets. Then your $171/day is concentrated enough for Meta to actually learn.
